## Problem Statement

Build `sinyuk-cli` as a TypeScript-based personal terminal workbench that replaces
fragmented Shell/Python scripts with a config-driven, interactive, and
feature-based system.

The project is not trying to become "a CLI with more commands" and it is not an
AI chat shell. The target is a long-lived terminal UI platform that combines the
execution power of scripts with GUI-like interaction inside the terminal.

The starting pain is clear:
- Existing scripts only accept parameters and break the interaction loop.
- Logic is scattered across files and tools, which makes reuse and maintenance
  hard.
- Environment rules and credentials tend to get hardcoded into scripts instead
  of living in explicit config.

## What Makes This Cool

The interesting version of this project is not "a nicer script runner." It is a
terminal workbench with two equally valid entry styles:

- Open the workbench with `sinyuk-cli` and navigate by feature.
- Jump straight into a feature command like `sinyuk-cli gitlab auto-mr --dry-run`.

That combination is the "whoa" factor: terminal-native automation with visible
state, guided interaction, config awareness, previews, progress, and logs,
without forcing every workflow back into a GUI or every fast path back into a
shell script.

## Constraints

- Phase 1 is about proving the architecture skeleton, not shipping real GitLab,
  image, or network automations.
- Feature-based architecture is mandatory. Avoid `commands/` + `utils/` as the
  main organizing idea.
- The platform should own only essential horizontal capabilities: config loading,
  logging, and graceful shutdown. Leverage Clipanion's built-in command routing
  and context; leverage Ink's app lifecycle and `useApp()`/`waitUntilExit()`;
  leverage Node's native `AbortSignal` for child process cancellation.
- Config precedence should be explicit:
  `CLI flags > project config > global config > defaults`.
  Config merge policy: **project config wins on conflict** (simple atomic
  replacement per key, no deep merge). The platform resolves that cascade once,
  then passes a read-only config snapshot into feature execution.
- Each feature should own its own schema, command surface, local UI state, and
  runtime behavior, but it should still expose a small unified entry contract so
  the platform knows how to discover and invoke it.
- Workbench and direct command entry should be treated as two routes into the
  same feature logic, not as two separate implementations.
- UI state belongs to the feature. Phase 1 should not introduce a global UI
  state machine.
- Each feature execution should get an isolated execution context containing at
  least a run ID, config snapshot, env snapshot, scoped logger, entry mode, and
  abort signal.
- Global config lives in `~/.config/sinyuk/config.yaml`.
- Project config lives in `./sinyuk.yaml`.
- Platform performs only structural validation on config load; each feature
  validates its own config section at execution time. This prevents unrelated
  feature config errors from blocking workbench startup.
- Validation should be explicit with YAML + Zod.
- Logs should be structured JSON under `~/.sinyuk-cli/logs/`.
- `SIGINT` must close resources cleanly and stop active work without leaving
  dangling state. Use Node's native `AbortController` and pass `AbortSignal` to
  Ink and child processes.
- Distribution matters because the output is a CLI artifact. Local development
  uses `npm link`; release path should support `npm install -g sinyuk-cli`.
- Dry-run is required for side-effecting features, but it should remain a
  feature concern rather than a global lifecycle contract. Dry-run and real
  execution should still share the same core logic path so preview behavior does
  not drift from real behavior.
- Interactive fallback policy: only fall back to interactive Ink flow when
  **both** conditions are met: (1) the process is running in a real TTY, and
  (2) the error is a recoverable missing-input case. Non-TTY / CI environments
  must fail fast with a clear error message. This prevents scripts from silently
  hanging on interactive prompts.
- Out of scope for this phase: real feature implementations, plugin market, AI
  integrations, permissions, and distributed execution.

## Premises

1. `sinyuk-cli` should be treated as a terminal workbench whose CLI is the entry
   shell, not as a pile of loosely related commands.
2. The platform boundary should stay horizontal: menu, config, logging,
   startup/shutdown, and routing belong to the platform.
3. Every feature should expose a small common entry contract such as
   `getEntry()` or `run(context)`. This is a discovery boundary, not a shared
   business workflow.
4. Feature execution semantics should stay local to the feature. The platform
   should not enforce a shared lifecycle like `collect -> plan -> preview ->
   execute` across all features.
5. Workbench and direct command entry should reuse the same feature execution
   logic rather than branching into separate implementations.
6. UI state belongs to the feature. Phase 1 should not introduce a global UI
   state machine.
7. Direct command entry is part of the product, not a future add-on. The user
   should be able to open the workbench or jump straight into a feature command.
8. Orchestration should live in the feature by default, but the platform may
   later extract clearly repeated horizontal capabilities once real overlap
   exists.
9. Platform-like capabilities such as task center, background sessions, or
   thread scheduling are valid future platform concerns, but they should not be
   forced into Phase 1 before there is evidence that multiple features need
   them.

## Approaches Considered

### Approach A: Workbench Shell Only

Build only the persistent Ink workbench in Phase 1. Features are entered from
the menu, and direct command entry is deferred.

Pros:
- Fastest path to a visible workbench.
- Keeps initial file count and architecture pressure low.
- Good fit for proving the Hello World pipeline.

Cons:
- Delays a core use case the project already wants: direct feature execution.
- Forces later routing work when feature commands are eventually added.
- Risks overfitting the architecture around menu-first usage.

### Approach B: Hybrid Shell + Feature Commands

Build both entry surfaces from the start:
- `sinyuk-cli` opens the workbench.
- `sinyuk-cli <feature> <action>` jumps directly to a feature command.

The platform stays thin. Features own their own logic, screens, and schemas.

Pros:
- Matches the intended product shape from day one.
- Keeps feature autonomy intact.
- Adds just enough shared contract to keep discovery, routing, and UX from
  fragmenting across features.
- Avoids inventing a global execution framework too early.
- Lets the workbench and direct commands coexist without duplicating business
  meaning at the platform layer.

Cons:
- Slightly more upfront routing work than a menu-only shell.
- Requires discipline to keep the shared entry contract thin and avoid turning
  it into a global feature lifecycle.

### Approach C: Shell + Task Center Substrate

Build Approach B, but also add a platform task center, session model, and
background job substrate in Phase 1.

Pros:
- Most "platform-like" in the long run.
- Sets up future long-running tasks and job history early.
- Could make the workbench feel more like a terminal desktop.

Cons:
- Too much infrastructure for an empty repository.
- Risks proving the task system before proving the first useful feature path.
- Increases Phase 1 complexity without evidence that it is yet needed.

## Recommended Approach

Choose **Approach B: Hybrid Shell + Feature Commands**.

This is the best balance between product truth and architectural restraint:
- It preserves the idea that the project is a workbench, not just a command set.
- It respects your correction that the platform should not standardize feature
  execution semantics.
- It keeps Workbench and direct command as two routes into the same feature
  logic, which avoids maintaining parallel implementations.
- It leaves space for future platform concerns such as task centers or session
  management without pretending they are needed on day one.

The shape of the system should look roughly like this:

```text
src/
  index.ts              # Clipanion CLI entrypoint
  platform/
    config/             # YAML load + simple merge (project wins)
    logging/            # JSON structured logger
    shutdown/           # thin AbortController wrapper
  shared/
    ui/                 # DEFERRED: extract when second feature needs reuse
      # Future components: spinner, progress, checkbox-list, status-line, tokens
    # shared/ui/ is the design system entry point.
    # Rule: New features MUST use shared/ui components before creating new ones.
    # If a new component is needed, add it to shared/ui/ so other features can reuse.
    # Phase 1: hello-world implements UI inline; extract to shared/ui when second
    # feature appears and reuse need is proven.
  features/
    index.ts            # explicit static registry
    hello-world/
      command.ts        # Clipanion command (thin adapter)
      screen.tsx        # Ink UI (thin adapter, uses shared/ui)
      schema.ts         # Zod schema for feature config
      run.ts            # canonical runner (all business logic here)
      index.ts          # FeatureEntry export
```

Key rule: `platform/` owns only config/logging/shutdown. `features/<name>/`
owns feature meaning. Clipanion handles command routing; Ink handles app
lifecycle. Do not wrap what is already provided.

This is a **thin platform + thick feature** architecture.

## Hello World Feature: Interaction State Table

| STATE | USER SEES | DETAIL |
|-------|-----------|--------|
| **LOADING** | Spinner + "Scanning {path}..." | Shows current directory being scanned |
| **EMPTY** | "No files found in {path}" | Warm message: "Try a different path, or check if the directory exists." + `[Change path]` action |
| **ERROR** | Red "Error: {message}" | Specific error (permission denied, path not found, etc.) + `[Retry]` action if recoverable |
| **SUCCESS** | Green checkmark + "Processed N files" | Summary line + option to `[View details]` |
| **PARTIAL** | Yellow warning + "Processed N of M files" | Lists failures below + `[Retry failed]` action |
| **SELECTING** | Checkbox list with file names | `Space` to toggle, `Enter` to confirm, `A` to select all |
| **PROGRESS** | Progress bar + "Processing file N of M..." | Shows current file name being processed |
| **DRY_RUN** | Cyan "Dry run:" prefix on all output | Makes it visually obvious no side effects are happening |

### Empty State Design Principles

Empty states are features, not afterthoughts. Each empty state must have:
1. **Context**: Why is this empty? (no files matching pattern, empty directory, etc.)
2. **Warmth**: Friendly tone, not a bare error
3. **Primary Action**: What can the user do next?

### Abort Handling

When user presses `Ctrl+C` during any state:
1. Current operation stops immediately via AbortSignal
2. Partial progress is shown (if any)
3. "Operation cancelled." message
4. Clean exit (no dangling state)

## Responsive & Accessibility

### Responsive Strategy: Rely on Ink, Don't Over-Engineer

1. **Use Flexbox, not hardcoded widths**: Use `<Box flexGrow={1}>` and `width="100%"`,
   never absolute character counts.
2. **Truncate over wrap**: Terminal line wrapping breaks alignment and causes flicker.
   Use `<Text wrap="truncate-end">` or `truncate-middle` for paths and descriptions.
3. **Single breakpoint**: If needed, only one "safety line" at 80 columns.
   Below 80: hide non-essential info (descriptions), keep core content (names).

```tsx
// Example: graceful degradation
const { columns } = useStdout();
const isNarrow = columns < 80;

<Box>
  <Text>{feature.name}</Text>
  {!isNarrow && <Text color="gray">{feature.description}</Text>}
</Box>
```

### Accessibility Strategy: Architecture is Accessibility

Screen readers struggle with interactive TUIs (cursor jumping, partial redraws).
Our answer: **the Hybrid architecture IS the accessibility solution.**

1. **CLI commands are the accessible mode**: Users who need screen readers or run
   in CI/headless environments use direct commands (`sinyuk-cli hello-world --path .`),
   not the interactive workbench.
2. **Plain text fallback**: Direct command output is linear console.log, not
   complex Ink panels. This is the accessibility path.
3. **No fake ARIA**: Don't try to add ARIA-like attributes to Ink components.
   The architecture already provides the accessible alternative.


The shared contract should stay intentionally small. Conceptually, each feature
should expose something like:

```text
FeatureEntry
  id
  metadata
  getCommand()          # returns Clipanion Command class
  getScreen()           # returns Ink component
  execute(executionContext)
```

The contract exists so the platform can discover and invoke features
consistently. It does **not** define how a feature internally models preview,
selection, confirmation, or execution.

Each feature must have a **canonical runner** (`run.ts`) that contains all
business logic. Both the Clipanion command and the Ink screen are thin adapters
that call this runner. This ensures DRY and keeps the two entry paths converged.

```text
CLI command.ts ----\
                    -> run.ts (canonical runner)
Ink screen.tsx ----/
```

The context boundary should also be explicit:

```text
ExecutionContext
  runId
  entryMode         # 'cli' | 'workbench'
  configSnapshot    # read-only, already merged
  envSnapshot
  logger            # scoped to this run
  abortSignal       # from platform shutdown controller
  dryRun
  isTTY             # for fallback policy
```

Feature code should receive context by injection, not by reaching into platform
singletons with global imports. That keeps features testable and keeps shared
side effects at the boundary.

Direct command and workbench entry must converge on the same feature execution
logic:

```text
CLI route ----------\
                      -> feature.execute(executionContext)
Workbench route ----/
```

If direct command input is incomplete but recoverable, the platform may hand off
to the feature's interactive UI instead of failing immediately. That fallback is
part of the Hybrid approach's value.

The first real proof should be a single `hello-world` feature that works in both
entry styles:
- Workbench path: `sinyuk-cli` -> menu -> hello world
- Direct path: `sinyuk-cli hello-world ...`

Inside that feature:
- UI state stays local to the feature.
- Dry-run and real execution share the same core logic path.
- Abort handling flows through the shared shutdown signal so `Ctrl+C` can stop
  side effects before Ink exits.

## Architecture Decisions (Scope Challenge)

The following decisions were locked during the Step 0 scope challenge. They
constrain Phase 1 implementation and should not be revisited without cause.

### AD-1: Thin Phase 1 Platform (1A)

**Decision**: Keep the Hybrid direction but shrink Phase 1 internal scaffolding.
Use Clipanion's built-in command routing and execution context directly.
Platform retains only thin config/logger/shutdown modules. `platform/cli`,
`platform/context`, and heavier wrappers are deferred until a second feature
proves their value.

**Rationale**: Clipanion already provides command path registration and context;
Ink already provides app lifecycle and `useApp()`/`waitUntilExit()`; Node
provides `AbortSignal` for child processes. Re-wrapping these in Phase 1 is
unnecessary complexity.

### AD-2: TTY-Only Interactive Fallback (2A)

**Decision**: Automatic fallback to interactive Ink flow only happens when:
1. The process is attached to a real TTY (`process.stdout.isTTY`).
2. The error is a recoverable missing-input case.

Non-TTY and CI environments fail fast with a clear error message.

**Rationale**: Prevents scripts, aliases, and CI jobs from silently hanging
when a parameter is missing. "Explicit > clever."

### AD-3: Canonical Runner per Feature (3A)

**Decision**: Every feature has exactly one canonical runner module (`run.ts`)
that contains all business logic. The CLI command and Ink screen are thin
adapters that call this runner.

**Rationale**: Guarantees DRY between the two entry paths. Makes testing
straightforward. Avoids "try to reuse but actually diverge" failure mode.

### AD-4: Simple Config Merge — Project Wins (4-simple)

**Decision**: When global and project config have conflicting keys, project
config wins (atomic replacement). No deep merge, no whitelist, no merge policy
complexity.

**Rationale**: Keep config behavior predictable. Deep merge causes subtle bugs
("why did this old global field leak in?"). Simple is maintainable.

### AD-5: Lazy Feature Config Validation (5A)

**Decision**: Platform performs only structural validation on config load.
Each feature validates its own config section when it executes, not at startup.

**Rationale**: Prevents one broken feature config from blocking the entire
workbench. Supports incremental feature development.

### AD-6: Explicit Static Feature Registry (6A)

**Decision**: Phase 1 uses an explicit static registry (`features/index.ts`).
Each feature is manually imported and registered. No directory scanning or
convention-based auto-discovery.

**Rationale**: Boring but stable. Easy to debug "why isn't my feature showing
up." Upgrade to auto-discovery only when feature count justifies it.

### AD-7: No Implicit Config Fallbacks (7A)

**Decision**: Config is a contract. No dual-source behavior. If a config value
is needed, it MUST exist in the config file. Platform does NOT provide hidden
defaults.

**Error Handling Specification:**

| Scenario | Behavior | Message |
|----------|----------|---------|
| Global config missing | **Fail Fast** | "Missing global config at ~/.config/sinyuk/config.yaml. Run `sinyuk-cli init` to create." |
| Project config missing | **OK** | Project config is optional; global config is sufficient. |
| YAML syntax error | **Fail Fast** | Print parser error with line/column number. |
| Required field missing | **Fail Fast** | "[Config Error] 'gitlab.token' is required but missing in ~/.config/sinyuk/config.yaml" |
| Type validation failed | **Fail Fast** | "[Config Error] 'timeout' must be a number, got string at line 15" |

**Implementation Rule:**
- **Zod schemas MUST NOT use `.default()` for config fields.**
- If a sensible default exists, document it in a config template, not in code.
- Config values come from ONE source: the config file. Never from code fallbacks.

**Rationale**: "Schrödinger's config" — when debugging, you can never know if a
value came from the user's file or a hidden code default. Explicit > clever.
